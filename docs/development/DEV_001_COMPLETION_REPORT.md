## 🏗️ 技术架构特点

### 模块化设计
```
TabletopCompanion (主入口)
├── MainController (主控制器)
├── Logger (日志系统)
├── ErrorHandler (错误处理)
├── ModuleBase (模块基类)
└── UIManager (UI管理器)
```

### 事件驱动架构
- 系统间通信基于事件机制
- 解耦的模块间交互
- 统一的事件处理和分发

### 配置管理
- 分层配置系统 (默认 → 保存 → 运行时)
- 自动配置合并和验证
- 模块级配置隔离

### 数据持久化
- `script_state` 基础的数据保存
- JSON序列化和反序列化
- 自动保存和恢复机制

## 🚀 关键技术决策

### 1. TTS API兼容性
- **决策**: 基于验证的API能力设计架构
- **原因**: 确保在TTS环境中的稳定运行
- **影响**: 所有模块都考虑TTS沙箱限制

### 2. 模块加载策略
- **决策**: 使用require机制加载模块
- **原因**: 简化模块依赖管理
- **影响**: 需要正确的模块路径配置

### 3. UI系统选择
- **决策**: 使用TTS原生XML UI系统
- **原因**: 最大兼容性和性能
- **影响**: UI设计受TTS限制但功能完整

### 4. 错误处理策略
- **决策**: 统一的错误分类和恢复机制
- **原因**: 提升用户体验和系统稳定性
- **影响**: 所有模块都需要遵循错误处理规范

## 📝 遇到的技术挑战

### 1. 模块循环依赖
- **问题**: MainController和其他模块间的循环引用
- **解决**: 使用全局变量和延迟初始化
- **结果**: 成功解决依赖问题

### 2. TTS环境模拟
- **问题**: 测试环境下缺少TTS API
- **解决**: 创建完整的TTS API模拟器
- **结果**: 测试可以独立运行

### 3. UI事件绑定
- **问题**: XML UI中的事件处理函数注册
- **解决**: 使用全局函数和UIManager代理
- **结果**: 事件处理正常工作
