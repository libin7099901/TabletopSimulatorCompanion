# 产品需求文档 - 桌游伴侣 (Tabletop Companion)

## 1. 产品概览
**产品名称**: 桌游伴侣 (Tabletop Companion)
**产品类型**: Steam创意工坊Mod (Tabletop Simulator)
**目标用户**: Tabletop Simulator (TTS) 玩家，尤其是那些希望简化规则查阅、自动化计分、获得智能规则辅助，并愿意贡献和分享自定义模板的玩家。
**核心价值主张**: 作为Tabletop Simulator的辅助Mod，通过深度集成TTS内部机制，提供高度可定制的智能规则查询、自动化计分和游戏流程引导，大幅提升线上桌游体验。

## 2. 问题陈述与解决方案
**核心问题**:
1.  规则查阅效率低下：在TTS中游玩桌游时，玩家查阅规则细节需依赖外部文档或记忆，频繁打断游戏流程和沉浸感。
2.  游戏状态管理繁琐：复杂桌游的回合记录、资源管理、分数统计等手动操作效率低、易出错，且缺乏自动化提示。
3.  TTS脚本开发门槛高：TTS内置的Lua脚本功能强大，但普通玩家难以利用其实现通用且智能的游戏辅助，限制了玩家自制内容的深度和广度。
4.  语言障碍：部分TTS游戏Mod或其规则手册仅提供英文或其他非母语版本，对非母语玩家造成理解障碍，影响游戏体验。
**现有解决方案的不足**:
1.  TTS自带工具：笔记本、计算器等功能单一，无法与游戏规则深度结合，不具备智能判断能力。
2.  现有TTS游戏Mod：通常针对特定游戏设计，通用性差，难以跨游戏复用辅助逻辑。
3.  外部辅助应用：无法直接获取TTS内的实时游戏状态，难以实现深度融合和自动化交互。
4.  人工翻译：玩家需手动查词或使用翻译工具，效率低下，且难以针对游戏特定术语进行优化。
**我们的解决方案**:
开发一个通用的、高度可定制的Tabletop Simulator Mod，通过加载外部或Mod内置的游戏模板文件（包含规则、UI定义、游戏逻辑等），来实现以下功能：
1.  智能规则查询：集成大语言模型（LLM），能够实时获取TTS内的游戏状态作为上下文，并理解游戏规则文档，回答玩家在游戏中的疑问。
2.  自动化计分与状态辅助：根据游戏模板定义的复杂计分规则，辅助玩家输入计分项并自动统计总分；实时更新关键游戏状态指示器（如回合数、玩家分数、资源数量等），减少人工管理负担。
3.  游戏流程引导：根据游戏模板定义，在Mod UI或TTS聊天中引导游戏流程，包括回合/阶段提示、玩家操作提示。
4.  高度可定制化与UGC生态：支持玩家（尤其是懂脚本的玩家）自制、分享和导入游戏模板，极大地扩展Mod的适用性和生命力。
5.  智能语言辅助：利用LLM提供游戏内容的自动翻译、指定术语翻译及本地存储功能，降低语言障碍。
**独特优势**:
1.  深度集成模拟器：直接通过Lua脚本与Tabletop Simulator的内部API（如游戏对象、UI、网络事件、游戏状态）进行交互，实现比外部App更精准、更实时的辅助和自动化。
2.  大模型智能辅助：充分利用LLM的能力，提供上下文感知、智能推理的规则查询和操作建议，并扩展到语言辅助功能。
3.  模板化、高度可定制性：提供一套标准化的游戏模板结构，让玩家可以通过修改模板文件（Lua脚本和JSON/XML配置）轻松适配不同的桌游，构建丰富的UGC生态。
4.  开源与社区驱动：项目将在GitHub上开源，鼓励社区参与和共同完善，不断丰富模板库和Mod功能。

## 3. 用户画像与使用场景
**主要用户画像**:
-   TTS核心玩家/Mod制作者 (A): 经常使用Tabletop Simulator进行各种桌游，对现有辅助工具不满足，希望为自己喜欢的游戏制作更强大的辅助Mod，愿意学习和使用Lua脚本来定义游戏模板。
-   TTS普通玩家 (B): 主要通过Tabletop Simulator与朋友进行线上桌游，希望有工具能简化规则查阅和计分，会通过创意工坊订阅并使用已有的"桌游伴侣Mod"及其兼容模板。
-   非母语玩家 (C): 游玩TTS上非母语的游戏时，希望获得实时、准确的翻译辅助，尤其是针对游戏特有术语的翻译。

**典型使用场景**:
-   场景1：Mod的加载与模板选择
    1.  玩家A（房主）在Steam创意工坊订阅"桌游伴侣Mod"。Mod在TTS启动时自动加载，并在TTS界面上显示一个自定义UI窗口。
    2.  玩家A在Tabletop Simulator中加载任意桌游（例如，一张空白桌子或某个游戏的官方Mod）。
    3.  Mod自动识别当前游戏的名称或ID (通过 `getLoadedObjects()` 或类似API分析场景内容，或读取 `getWorkshopInfo()` 等)，并在其UI的模板选择列表中将匹配的模板置于最前列。
    4.  如果玩家A在Mod设置中启用了"自动加载模板"选项，Mod将在识别到匹配模板后自动加载它。否则，玩家A需手动选择并加载模板。
    5.  Mod根据模板定义，开始在UI中显示回合数、玩家分数、资源等指示器，并在TTS聊天框或Mod UI中输出流程引导信息。
-   场景2：智能规则查询（基于游戏模板）
    1.  游戏进行中，玩家B对某个规则有疑问（例如，他手里的某张卡牌能否在当前回合使用）。
    2.  玩家B在Mod提供的自定义UI中输入问题，或在TTS聊天框内使用预设的指令（如"/rule 在当前阶段，我可以使用这张卡牌吗？"）。
    3.  Mod通过Lua脚本，实时获取TTS内的游戏状态（例如：当前回合数、玩家B的手牌 (可能是数量或类型，具体内容取决于游戏和模板设计，直接获取其他玩家私密手牌内容通常不可行)、桌面上的物品位置、玩家B的资源情况），并将这些状态信息连同模板中的游戏规则文档，一同作为上下文传递给LLM。
    4.  LLM返回回答，显示在Mod的自定义UI窗口或TTS聊天框中，提供精准的规则解释或操作建议。
-   场景3：自动化计分与状态更新
    1.  玩家A执行了一个操作（例如，移动了一个资源标记到一个得分区域，或点击了Mod UI上一个"获得2金币"的按钮）。
    2.  Mod的Lua脚本监听TTS的相应事件（如物体移动 `onObjectDrop`、自定义按钮点击 `UI.onClick` 或类似事件）。
    3.  Mod根据模板定义的game_logic和state_transitions，自动更新玩家A的分数指示器或聊天区显示"玩家A获得2金币"。
    4.  当游戏进入计分阶段，Mod的UI会引导玩家输入特定计分项（例如，每个玩家拥有的建筑物数量），然后Mod自动根据模板定义的计分规则计算总分，并将结果同步显示给所有玩家。
-   场景4：玩家自制/导入模板
    1.  玩家C（一位Mod制作者）想要为一款TTS上没有辅助功能的桌游制作"桌游伴侣"模板。
    2.  玩家C根据"桌游伴侣Mod"的模板规范，编写Lua脚本和JSON/XML配置文件，定义该游戏的规则、UI元素和自动化逻辑。
    3.  玩家C将制作好的模板文件放置到"桌游伴侣Mod"的指定模板目录中 (需验证TTS API是否允许Mod读取特定子目录的文件，如 `Mods/Saves/MyMod/templates/` 或依赖用户手动将模板内容粘贴到Mod的配置界面中，如果直接文件访问受限)，或通过Mod提供的导入功能加载。
    4.  玩家C在TTS中加载"桌游伴侣Mod"，然后选择并加载新制作的模板进行测试，并可能将模板分享到Steam创意工坊供其他玩家使用。
-   场景5：通用规则查询（无模板或手动上传规则书）
    1.  玩家D在TTS中游玩一个尚未有"桌游伴侣Mod"模板支持的游戏，或希望使用自定义规则书。
    2.  玩家D通过Mod UI的通用规则查询入口，或在TTS聊天框内使用通用指令（如"/ask [你的问题]"）。
    3.  此外，玩家D还可以直接在TTS桌面上对相关游戏元素（如卡牌、棋子、规则书对象等）进行右键操作，并在弹出的上下文菜单中选择"询问规则"选项，Mod将自动提取该元素的相关信息作为上下文。
    4.  Mod将尝试从当前加载的TTS游戏Mod中查找对应的规则手册 (例如，通过分析游戏对象的文件名或描述，或查找常见的规则书对象GUID)。如果Mod内未找到或玩家希望使用其他版本，玩家D也可以手动上传一份该游戏的规则文档（PDF/TXT/Markdown、JPG/PNG等图片格式）。对于PDF，Mod可能需要用户先将其转换为文本或图片，或者LLM服务本身支持PDF解析。 对于图片格式的规则文档，Mod将利用OCR技术提取文本 (如果LLM不支持直接图片输入，且在TTS环境中直接运行OCR库不可行，可能需要提示用户使用外部OCR工具或上传文本)，或直接将图片数据传递给支持多模态的LLM进行解读。
    5.  Mod将玩家的问题和找到/上传的规则文档作为上下文传递给LLM。
    6.  LLM返回回答，显示在Mod的自定义UI窗口或TTS聊天框中，提供基于规则文档的解答。由于此模式下不包含TTS内的实时游戏状态上下文，LLM的回答将是理论性的规则解释。
-   场景6：游戏内容翻译
    1.  玩家E在TTS中加载了一款只有英文文本的桌游。
    2.  玩家E在Mod UI中选择目标语言（如中文），并选择"自动翻译游戏内容"功能。
    3.  Mod通过Lua脚本获取TTS桌面上可见的游戏对象文本（如卡牌名称、描述、板图文字 ，需要调研API获取这些文本的具体方法，如 `object.getName()`, `object.getDescription()`, `object.getGMNotes()`，或解析自定义对象的脚本数据）。
    4.  Mod将这些文本发送给LLM进行翻译。
    5.  翻译结果显示在Mod UI中（例如，作为悬浮提示、独立翻译面板或在聊天框中显示），如果TTS API允许，则尝试直接更新游戏对象上的文本 (需重点验证 `object.setName()`, `object.setDescription()` 等函数是否能修改非本Mod创建的对象的这些属性，或者对象是否有可供脚本修改的文本字段。备选方案包括使用3D UI Text元素覆盖在原对象上，或对于支持自定义贴图的对象，动态生成包含翻译文本的贴图并更新)；对于图片格式的游戏内容，翻译结果将通过Mod UI叠加层、悬浮提示或独立翻译面板等方式进行展示，以确保玩家能看到翻译内容。
    6.  Mod将翻译结果存储在本地缓存 (利用 `Global.script_state` 或对象的 `script_state`，需注意大小限制；如果API允许，则写入Mod的本地文件)，下次加载同一游戏时，已翻译内容可快速显示。
    7.  玩家E遇到一个游戏中的特殊术语（如"Victory Points"），手动在Mod UI中输入该术语，并请求翻译为"胜利点"。Mod将此术语翻译并存储，下次遇到时自动识别并显示翻译。

## 4. 功能需求
**核心功能 (MVP必需)**:
1.  Mod加载与基础UI:
    *   Mod作为Tabletop Simulator的通用加载项，在TTS启动时自动加载，并在TTS界面上显示一个可拖动、可调整大小的自定义UI窗口。
    *   该UI窗口应包含用于加载规则手册、配置LLM服务、查看帮助信息等基础控件。未在MVP版本中实现的功能（如自动化计分、游戏流程引导、游戏内容自动翻译等）在UI中应有占位元素，但不可点击或操作 ，并明确提示"功能开发中"。
    *   当TTS房间中加载或切换游戏时，Mod应自动识别当前游戏的名称或ID，并在其规则手册选择列表中将匹配的规则手册（如果存在）置于最前列。
    *   Mod应提供一个可选的"自动加载规则手册"设置。如果该设置启用，Mod将在识别到匹配规则手册后自动加载它。
    *   当规则手册未加载时，显示"加载规则手册"按钮；当规则手册已加载时，显示"更换规则手册"和"卸载规则手册"按钮。Mod UI 应能反映从本地服务器获取的规则状态。
2.  规则手册加载与处理 (通过本地HTTP服务):
    *   MVP版本中，Mod将专注于通过本地HTTP服务 (`local_rules_server.py`) 读取和处理游戏规则手册。用户在Mod UI中提供规则文件的本地路径 (支持图片如JPG/PNG，文本如TXT/Markdown) 或直接输入文本规则。
    *   **文件路径加载**: Mod将文件路径发送给本地服务器的 `/load_rules` 端点。服务器负责读取文件，如果是图片则转换为Base64编码，并存储在服务器内存中。
    *   **文本直接加载**: Mod将文本内容发送给本地服务器的 `/load_rules` 端点。服务器直接存储文本。
    *   本地服务器应优先处理图片规则。如果同时尝试加载图片和文本，以最后成功的为准，或由服务器策略决定（当前实现为后者覆盖前者）。
    *   Mod本身不再直接处理文件读取或Base64转换，也不再直接处理OCR。这些由本地服务器和其调用的Ollama服务负责。
3.  智能规则查询（LLM集成，通过本地HTTP服务代理）:
    *   LLM服务配置与用户持久化：在Mod的UI中提供配置外部LLM服务API的入口（如Ollama API地址、API Key等，尽管当前本地服务器直接使用硬编码或环境变量的Ollama地址）。这些配置信息应支持用户自定义并持久化存储在Mod的 `script_state` 中。Mod在与本地服务器通信时，会将相关模型参数（如模型名称）传递给本地服务器的 `/ollama_proxy` 端点。
    *   LLM行为设定：Mod应提供配置LLM系统提示词（system_prompt）的接口，用于定义LLM的角色和行为规范。此提示词将作为请求的一部分发送给本地服务器。
    *   玩家提问入口：玩家可以在Mod提供的自定义UI输入框或TTS聊天框内使用特定指令向LLM提问。Mod也应支持右键菜单查询（待适配新架构）。
    *   **请求代理**：Mod将用户的问题、游戏上下文（如果适用）、系统提示词以及用户配置的LLM模型名称，发送给本地服务器的 `/ollama_proxy` 端点。
    *   本地服务器接收到请求后，会检查其内存中是否有已加载的图片规则（Base64）。如果有，则将图片数据附加到向Ollama服务发送的最终请求中。文本规则已由Mod整合到发送给代理的prompt中。
    *   本地服务器随后将完整的请求（可能包含图片）转发给实际的Ollama服务，并将Ollama的响应返回给Mod。
    *   回答展示：LLM的回答应显示在Mod的自定义UI窗口或TTS聊天框中。
    *   通用规则查询模式：在此模式下，规则通过本地服务器加载（文件路径或文本），查询同样通过本地服务器代理。
4.  Mod状态持久化（基础）:
    *   MVP版本应实现基础的Mod状态持久化，主要包括：
        *   LLM服务配置：用户配置的LLM API地址 (如果将来不由本地服务器固定) 和模型名称等信息，存储在 `script_state`。
        *   Mod UI状态：如输入框内容等。
    *   规则手册数据本身**不再由Mod直接持久化**。规则的持久化由用户负责（即文件本身），Mod启动时或用户操作时，通过本地服务器按需加载。本地服务器仅在内存中维护当前会话的规则。
5.  MVP版本与完整版本的兼容性:
    *   MVP版本的设计和实现应充分考虑未来完整版本的扩展性。所有接口和数据结构应具备良好的兼容性，确保在后续迭代中，现有MVP功能能够平滑升级并与新增功能无缝集成。例如，规则手册的数据结构应能轻松扩展以支持游戏模板的更复杂定义。

**增强功能 (后续版本)**:
1.  游戏模板解析与执行:
    *   Mod能够解析并执行由玩家制作的游戏模板文件。模板文件应以结构化的格式（如JSON/XML）定义游戏信息、UI布局、LLM提示词，并包含Lua脚本文件来定义具体的游戏逻辑。
    *   支持从Mod目录中的预设路径加载模板，或通过Mod UI实现模板文件的导入（需重点调研TTS API对文件系统的访问能力，包括沙箱内特定路径的读写权限，以及是否能触发标准的文件选择对话框。备选方案包括用户粘贴模板内容或通过URL加载模板）。
2.  智能规则查询增强（基于模板的上下文获取）:
    *   实时上下文获取：Mod的Lua脚本能够通过TTS的API，实时获取当前游戏状态数据作为LLM查询的上下文，包括但不限于：
        *   当前回合/阶段信息。
        *   所有玩家的可见分数、资源数量。
        *   桌面上的特定对象（卡牌、棋子、指示物）的位置、状态、归属。
        *   当前活跃玩家信息。
        *   部分可通过脚本推断的隐藏状态（例如玩家手牌数量而非具体内容）。
    *   支持LLM的思考功能/推理能力，通过精心设计的提示词，引导LLM在回答复杂问题时进行逻辑推理和步骤分解。
3.  游戏流程引导与状态显示:
    *   流程引导：根据模板定义的phases和turns，在Mod UI或TTS聊天框中实时输出当前游戏阶段、轮到哪个玩家操作等引导信息。
    *   状态指示器：在Mod的UI上显示由模板定义的关键游戏状态指示器（如回合数、玩家分数、资源数量、特定标记状态等）。这些指示器应能根据game_logic中的state_transitions实时更新，并且其值应同步给房间内所有玩家。
    *   操作完成/推进按钮：Mod UI提供"完成操作"、"回合结束"等按钮，玩家点击后触发Mod内定义的流程推进逻辑。
4.  自动化计分辅助:
    *   计分逻辑执行：Mod能根据模板中scoring_rules定义的复杂计分逻辑，自动计算玩家得分。
    *   计分UI引导：在计分阶段，Mod的UI可以引导玩家输入特定计分项（例如，手牌分数、区域控制分），然后Mod自动汇总并显示总分。
    *   实时分数更新：在游戏进行中，如果模板定义了操作触发的分数变动，Mod能即时更新玩家的分数指示器。
5.  玩家操作触发器与脚本交互:
    *   Mod的Lua脚本能监听TTS内的核心事件，例如：
        *   特定游戏对象（卡牌、棋子）的移动、翻转、吸附。
        *   自定义按钮的点击。
        *   玩家加入/离开房间。
        *   Mod自定义UI中的交互事件。
    *   这些事件可以触发game_logic中定义的游戏逻辑，实现自动化更新状态、计分或流程推进。
6.  语言辅助功能（LLM驱动）:
    *   游戏内容自动翻译：
        *   Mod应能利用LLM对TTS游戏中的可见文本内容（如卡牌描述、棋子文本、板图上的文字、自定义UI文本等）进行自动翻译。
        *   翻译结果显示在Mod UI中（例如，作为悬浮提示、独立翻译面板或在聊天框中显示），如果TTS API允许，则尝试直接更新游戏对象上的文本 (如前所述，这是关键验证点。如果直接修改不可行，备选方案包括：1. 使用TTS的3D UI Text元素创建浮动文本覆盖在原对象附近；2. 对于支持自定义贴图的对象，如卡牌，动态生成包含翻译文本的新贴图并应用；3. 在Mod的专属UI面板中集中展示选中对象的翻译)；对于图片格式的游戏内容，翻译结果将通过Mod UI叠加层、悬浮提示或独立翻译面板等方式进行展示，以确保玩家能看到翻译内容。
        *   翻译结果应支持本地存储（基于游戏Mod或模板的ID），以便下次加载同一游戏或模板时，已翻译内容可快速调用，无需重复进行LLM翻译。存储时同样需注意 `script_state` 的大小限制和文件访问权限。
    *   指定术语翻译与管理：
        *   玩家可以手动指定游戏中的特定术语（如"Victory Points"、"Action Points"、"Phase X"）进行翻译，并将其存储为自定义词汇表。
        *   Mod应能识别这些自定义术语，并在自动翻译或规则查询时优先使用玩家指定的翻译。
        *   提供Mod UI界面供玩家管理（添加、编辑、删除）这些自定义术语。
    *   目标语言选择：提供Mod UI界面供玩家选择翻译的目标语言。
7.  Mod状态持久化（完整版）:
    *   Mod的状态（如已加载模板、LLM配置、当前游戏辅助进度、本地翻译缓存、自定义术语表）能够随TTS游戏存档一同保存和加载，确保下次加载游戏时能恢复之前的辅助状态。
    *   持久化存储机制需考虑TTS Modding API对本地文件系统访问的限制，可能存储在Mod的特定数据文件或TTS的全局脚本缓存中。对于大量数据，应考虑数据结构优化和可能的压缩（如果Lua支持相关库且性能允许）。
8.  游戏模板多语言支持:
    *   游戏模板应支持多语言文本定义，Mod能根据用户在Mod UI中设置的语言偏好加载相应语言的规则和UI文本，从而为不同语言的玩家提供本地化体验。
    *   模板制作者可以在模板文件中提供不同语言的llm_rules_document、player_rule_book、ui_definition中的文本内容等。
9.  模板创作辅助工具：提供外部桌面应用或TTS内更高级的Mod辅助工具，简化非脚本开发者制作游戏模板的流程（如可视化UI编辑器、规则提取工具）。
10. 模板共享与发现增强：在Mod UI中直接集成创意工坊的模板浏览和下载功能，方便玩家发现和加载其他制作的模板。
11. 语音输入/输出：集成语音识别和TTS的语音输出功能，实现更自然的交互。

**用户交互流程**:
*   **Mod初始化与模板加载**：
    1.  用户订阅Mod，Mod在TTS启动时加载，显示主UI。
    2.  用户在TTS中加载游戏。
    3.  Mod尝试自动识别游戏并推荐/自动加载匹配的模板/规则手册。
    4.  用户可手动选择、导入或上传模板/规则手册。
    5.  模板加载后，Mod UI根据模板定义更新（状态指示器、流程引导）。
*   **智能规则查询**：
    1.  用户通过Mod UI输入框或聊天指令提问。
    2.  用户可右键点击游戏元素选择"询问规则"。
    3.  Mod获取游戏状态（若有模板）和规则文档作为上下文。
    4.  上下文发送给LLM。
    5.  LLM回答显示在Mod UI或聊天框。
*   **自动化计分与状态更新（基于模板）**：
    1.  玩家执行游戏操作（如移动对象、点击Mod UI按钮）。
    2.  Mod脚本监听TTS事件或UI事件。
    3.  Mod根据模板逻辑更新分数、资源等状态指示器。
    4.  计分阶段，Mod UI引导玩家输入计分项，自动计算并显示总分。
*   **游戏流程引导（基于模板）**：
    1.  Mod根据模板定义，在UI或聊天框提示当前阶段、玩家回合。
    2.  玩家可通过Mod UI按钮（如"回合结束"）推进游戏流程。
*   **内容翻译**：
    1.  用户在Mod UI选择目标语言并启用翻译功能。
    2.  Mod获取可见游戏对象的文本。
    3.  文本发送给LLM翻译。
    4.  翻译结果通过UI叠加、面板或直接修改（若API允许）显示。
    5.  用户可管理自定义术语表，优化翻译。
    6.  翻译结果本地缓存。

## 5. 非功能性需求
**性能要求**:
*   流畅性：Mod运行应平滑，不导致Tabletop Simulator的帧率显著下降或出现卡顿，即使在处理复杂游戏状态、LLM交互和大量文本翻译时。避免在Lua的 `update()` 函数或高频事件回调中执行耗时操作（如LLM网络请求、复杂数据处理）；使用计时器 (`Timer`) 或协程 (`coroutine`) 分散任务。
*   响应时间：Mod UI的交互、状态指示器更新应即时响应；LLM查询和翻译请求的响应时间主要取决于LLM服务本身及网络状况，但Mod内部数据处理应高效。对于LLM请求，应有明确的加载/等待提示，并允许用户取消长时间的请求。
*   资源占用：Mod应优化Lua脚本的执行效率和内存占用，避免对TTS的整体性能造成负担。翻译结果的本地存储应高效管理，避免占用过多磁盘空间。定期检查和优化数据结构，避免内存泄漏。对于大型文本缓存，考虑LRU (Least Recently Used) 等淘汰策略，如果存储空间受限。
**安全要求**:
*   LLM服务配置安全：LLM API密钥等敏感信息应安全存储（如使用TTS的Mod本地缓存或加密存储）。用户在配置LLM服务时需明确同意其游戏状态数据和待翻译文本可能被发送到外部LLM服务提供商。由于TTS的 `script_state` 通常是明文存储在存档中的，应明确告知用户此风险。不建议Mod本身实现复杂的加密算法，可考虑让用户每次启动时输入密钥，或依赖TTS未来可能提供的更安全的凭证管理机制。严禁在开源代码中硬编码任何API密钥。
*   模板安全性：由于模板包含可执行的Lua脚本，存在执行恶意代码的风险。虽然TTS提供了一定程度的沙箱机制，但Mod本身应倡导社区审查机制，并在文档中提醒用户谨慎使用未知来源的模板。可以考虑在Mod内对加载的模板脚本进行基础的静态分析，检测已知的恶意模式或高危API调用（如未经用户明确许可的网络请求），并向用户发出警告，但这实现复杂且效果有限。主要依赖社区和用户警惕性。
*   数据隐私：除了为LLM查询和翻译发送必要的游戏状态上下文数据和待翻译文本外，Mod不应收集或传输任何用户的个人身份信息。
*   脚本错误处理：Mod的脚本应具备健壮性，避免未捕获的错误导致整个TTS崩溃。使用 `pcall` 或 `xpcall` 保护关键的函数调用，特别是与外部服务交互或执行模板脚本时。
**可用性要求**:
*   易用性：Mod的订阅安装过程应简单明了（通过Steam创意工坊一键订阅）。Mod在TTS内的加载过程应直观（通过TTS的Modding菜单加载）。自定义UI设计应直观，易于理解和操作。翻译功能应有清晰的触发方式和结果展示。提供清晰的入门教程和FAQ。
*   兼容性：Mod应与主流版本的Tabletop Simulator保持兼容。在文档中明确支持的TTS版本范围。
*   健壮性与错误处理：
    *   Mod加载异常：当Mod加载失败或规则手册/模板解析错误时，应在TTS聊天框或Mod UI中提供明确的错误提示 （包括错误原因和可能的解决步骤），而不是默默失效或导致TTS崩溃。
    *   LLM服务异常：当LLM服务不可用、响应超时或返回错误时，Mod应提供友好的错误提示 （例如："LLM服务连接失败，请检查您的网络和API配置。"），并建议用户检查配置或稍后重试。特别是翻译失败时，应有明确提示。
    *   OCR/多模态解读失败：当图片规则手册无法被OCR准确识别或多模态LLM无法有效解读时，应提供明确提示并引导用户尝试其他方式（如手动输入文本、上传更高质量图片 ，或提供一个简单的文本编辑区域供用户修正OCR结果）。
    *   脚本运行时错误：Mod脚本应尽量捕获并处理运行时错误，防止其影响TTS的正常运行，并在开发者控制台中输出详细的错误日志 (对于用户，可以提供一个复制错误信息给开发者的按钮)。
    *   规则手册/模板内容校验：在加载规则手册或模板时进行基础的结构和数据类型校验，防止格式错误的输入导致Mod崩溃。
    *   用户误操作防护：关键操作（如删除规则手册/模板、重置游戏 、清空API Key配置）应提供二次确认机制。
*   容错性：对于用户误操作、网络连接中断或异常情况，应有友好的提示和恢复机制。
**扩展性要求**:
*   游戏模板兼容性：Mod应设计灵活的模板结构和解析引擎，以支持未来更复杂、功能更丰富的桌游规则，并支持模板版本号管理。
*   LLM服务扩展：易于集成未来可能出现的其他LLM服务提供商或模型。通过配置文件或UI让用户选择不同的LLM预设或自定义API端点。
*   TTS版本兼容：考虑TTS未来版本更新可能带来的 API 变化，Mod应尽量保持向下兼容性或提供明确的升级指南。
*   翻译语言扩展性：支持未来增加更多目标翻译语言。语言列表应可配置或从外部源更新。

## 6. 技术考虑
**建议技术栈**:
-   前端: Tabletop Simulator 自定义UI (基于XML/Lua)
-   后端: **本地Python HTTP服务 (`local_rules_server.py`)**，负责规则文件处理、Base64转换（图片）、规则临时存储，并作为TTS Mod与Ollama服务之间的代理。LLM通过此本地服务进行HTTP API调用。
-   数据库: 无独立数据库，Mod状态持久化依赖TTS存档机制 (`script_state`)。规则文件由用户本地保存。
-   其他: Lua (TTS Modding), Python (Flask, Flask-CORS, requests), JSON/XML (模板配置)
**技术约束**:
*   **本地HTTP服务依赖**: Mod的核心功能（规则加载、LLM查询）现在依赖于用户在本地成功运行 `local_rules_server.py`。
*   TTS Modding API限制：Mod的功能实现仍受限于Tabletop Simulator提供的Modding API的能力和安全沙箱。`WebRequest` 用于与本地服务器通信。
*   Lua脚本性能：Lua脚本的执行效率需要被仔细优化，以避免对游戏性能造成负面影响。
**集成需求**:
*   LLM API: 通过本地 `local_rules_server.py` 代理集成各种LLM服务的HTTP API (当前主要为Ollama)。
*   本地HTTP服务 API: TTS Mod 需要与 `local_rules_server.py` 定义的API端点 (如 `/load_rules`, `/get_current_rules`, `/clear_rules`, `/ollama_proxy`) 进行稳定通信。
*   Tabletop Simulator API集成：
    *   游戏对象交互：利用TTS API获取和操作游戏对象的状态。
    *   自定义UI：利用TTS内置的UI系统。
    *   事件系统：监听TTS内部事件。
    *   网络事件：确保Mod的状态和UI更新在所有玩家间保持一致 (对于规则加载状态，主要由发起操作的玩家与本地服务器交互，然后可以将状态通知其他玩家)。
    *   **文件系统访问**: Mod本身不再尝试直接访问文件系统。此责任转移到 `local_rules_server.py`。
*   游戏模板结构定义 (完整版本):
    游戏模板将是本Mod的核心，其结构应包含但不限于以下内容：
    *   `template_version`: 模板版本号，用于兼容性管理。
    *   `game_info`: 游戏基本信息，如名称、作者、描述、支持人数范围。
    *   `llm_prompts`:
        *   `system_prompt`: 提供给LLM的系统级提示词，用于定义其角色（例如"游戏规则专家"、"公正的裁判"、"翻译助手"）和整体行为规范（包括回答的准确性、简洁性、实用性、中立性等）。
        *   `user_query_template`: 用户提问时，如何构建完整的问题（包含TTS实时上下文）。
    *   `llm_rules_document`: 详细的游戏规则文本，供LLM学习和理解。
    *   `player_rule_book`: 供玩家阅读的规则说明书（可支持Markdown格式）。
    *   `ui_definition`: 定义Mod在TTS内的自定义UI元素布局、样式和行为（基于TTS的UI系统），包括：
        *   指示器（indicators）：如回合、分数、资源等，及其初始值、更新规则、显示位置/样式。
        *   按钮（buttons）：如"完成操作"、"重置"、"后退"等，及其文本、行为、绑定的Mod脚本函数。
        *   输入框（input_fields）：用于玩家输入分数等。
        *   自定义组件（custom_components）：如果需要更复杂的定制UI。
    *   `game_logic_script`: 核心游戏逻辑，用Lua编写。这是模板的核心，将：
        *   监听TTS事件，如对象移动、按钮点击。
        *   根据规则处理游戏状态转换（state_transitions）。
        *   更新Mod UI上的指示器。
        *   触发LLM查询请求（传递TTS实时状态）。
        *   实现复杂的计分规则（scoring_rules）。
    *   `resources`: Mod内部使用的媒体资源引用，如Mod内图片路径、音效文件路径。
    *   `localization`: (可选) 多语言文本定义，由模板制作者决定是否提供。

## 8. 成功指标与风险
**成功指标**:
-   创意工坊订阅量与用户评价：Mod在Steam创意工坊的订阅数量和平均评价。
-   游戏模板活跃度：玩家为不同游戏制作和分享的"桌游伴侣Mod"模板数量。
-   用户体验提升：用户反馈Mod在简化规则查阅、计分和流程引导方面的实际帮助，以及语言障碍的有效缓解。
-   开源社区参与度：GitHub项目的Star、Fork、Issue和Pull Request数量。
**主要风险**:
-   TTS Modding API限制：Tabletop Simulator的Modding API可能无法完全满足所有设想的功能（特别是对某些隐藏状态的访问或复杂UI的自定义，以及直接修改游戏对象文本的能力），需要进行功能裁剪或寻找变通方案。
    *   **应对策略**：在项目启动初期即进行彻底的API调研和原型验证（如里程碑规划中所述）。对于受限功能，设计并准备好降级方案。例如，若不能直接修改对象文本，则采用UI覆盖或侧边栏显示翻译。若文件访问受限，则更多依赖用户手动输入或 `script_state`。保持与TTS开发者社区的沟通，了解API的未来发展方向。
-   Lua脚本开发门槛与复杂性：对于不熟悉Lua和Modding的开发者来说，学习曲线较陡峭；管理复杂游戏逻辑的Lua脚本可能变得庞大和难以维护。
    *   **应对策略**：提供清晰的模板制作指南和丰富的示例；模块化脚本设计；考虑社区协作。鼓励使用良好的Lua编程实践，如编写单元测试（可以使用模拟TTS环境的外部Lua测试框架）。
-   LLM服务稳定性与成本：外部LLM服务的可用性和响应速度可能不稳定，且持续调用可能产生费用。特别是翻译大量文本可能导致较高的成本和延迟。
    *   **应对策略**：在Mod中提供LLM服务配置选项，允许用户自行选择和承担风险；优化LLM调用频率和上下文大小；实施翻译缓存机制，减少重复调用。提供明确的"清除缓存"和"重新翻译"选项。允许用户设置LLM请求的超时时间。
-   恶意模板脚本：由于模板包含可执行的Lua脚本，存在玩家上传恶意脚本的风险。
    *   **应对策略**：在创意工坊发布时依赖Steam的审查机制；在Mod文档中提醒用户谨慎使用未知来源模板；鼓励社区对模板进行审查。探索在Mod加载模板时，对脚本进行一些基础的权限声明或风险提示，例如告知用户此模板需要网络权限。
-   TTS版本更新兼容性：Tabletop Simulator本身的更新可能导致Mod API变化，影响Mod兼容性。
    *   **应对策略**：密切关注TTS更新日志，及时调整Mod代码；在发布时注明兼容的TTS版本。建立Mod的版本管理和发布流程，方便快速响应和修复兼容性问题。
-   翻译质量与幻觉：LLM的翻译质量可能不尽如人意，尤其是在游戏特有术语和语境理解上可能出现"幻觉"或不准确。
    *   **应对策略**：提供指定术语翻译功能让玩家修正；在LLM提示词中强调游戏语境；允许玩家切换不同的LLM服务。允许用户反馈不满意的翻译结果，以便改进提示词或未来可能的微调。
-   OCR准确性：图片格式规则手册的OCR识别准确性可能受图片质量、字体等因素影响，进而影响LLM的理解。
    *   **应对策略**：在OCR处理前进行图像预处理（如果可能）；允许玩家手动修正OCR结果；优先推荐使用文本格式的规则文档。如果依赖外部OCR服务或多模态LLM，明确告知用户其局限性。

3.  指导模板结构设计：基于上述分析，精确设计game_logic和ui_definition等模板字段的结构，确保其具备足够的灵活性和表达力，能够通用地描述各类游戏的复杂规则和交互界面。
4.  赋能底层引擎实现：通过对多样化游戏机制的透彻理解，指导Mod底层解析和执行这些模板的Lua脚本实现，确保其能够健壮、高效地处理各种规则逻辑和状态转换。
