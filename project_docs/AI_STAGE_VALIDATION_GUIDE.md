# **AI阶段任务验证与质量保证指南 (`AI_STAGE_VALIDATION_GUIDE.md`)**

**版本**: 1.2
**最后更新**: `{{YYYY-MM-DD}}`

**目的**: 本指南为 `OrchestratorAgent` 和所有专业AI角色提供一套标准的行为规范和检查清单，旨在确保AI在项目各阶段生成的任务、分配的任务以及最终的交付物都符合预设的质量标准、完整性要求，并且与项目需求、设计文档保持一致。

**重要提示：本指南定义的验证流程和检查清单，是在工作区根目录下 `.cursor/rules.md` 文件中定义的全局AI助手行为准则基础上的具体化和补充。`OrchestratorAgent` 负责整合这两部分，并优先应用 `project_docs/custom_validation_rules.md` (如果存在) 中定义的项目特定规则。**

## **核心验证原则**

1.  **文档驱动**: 所有AI行动和产出都必须基于并参考最新的项目文档（尤其是 `PROJECT_CONTEXT.md` 中索引的文档，包括但不限于需求、设计、架构、接口、命名规范等）。
2.  **需求对齐**: 确保AI的理解和执行始终与用户需求、功能规格说明（FSD）、用户故事以及"需求深化与澄清阶段"的结论对齐。
3.  **阶段门控**: 在每个关键阶段（任务生成、任务分配、任务完成、阶段结束，包括"UI逻辑先行"草案的评审）设立明确的检查点和质量门。
4.  **完整性优先**: 优先确保交付物的完整性，包括代码、文档（含 `PROJECT_SESSION_STATE.md` 的适时更新）、测试、配置等。
5.  **质量内建与客观评估**: AI角色在执行任务过程中就进行自我检查和质量控制。**在提交产出时，必须对照本指南及 `custom_validation_rules.md` 中的相关检查清单进行自我评估，并就每一项检查点的符合情况提供明确的理由或可验证的证据（例如，指向特定代码行、文档章节或测试结果）。**
6.  **持续反馈**: `OrchestratorAgent` 应就验证结果向相关AI角色或用户提供及时、明确的反馈。
7.  **自定义规则优先**: 逻辑保持不变（参考原始文档）。
8.  **严格评估**: 对于在需求或计划中标记为"重要"、"核心"或"必须实现"的功能点和质量标准，必须采用更严格的评估标准，确保100%实现和通过测试。

## **一、`OrchestratorAgent` 的阶段验证职责**

`OrchestratorAgent` 作为项目总指挥，在以下关键节点必须执行验证程序：

### **规范1: 需求深化与澄清阶段后的检查**
在"需求深化与澄清阶段"完成后，OrchestratorAgent必须检查：
```markdown
【需求深化验证检查清单】

✅ **需求明确性检查**:
- 确认所有原始需求中的模糊点已澄清
- 关键功能、非功能性需求已有明确定义
- 用户验收标准已初步定义

✅ **文档更新检查**:
- `PROJECT_CONTEXT.md` 是否已更新以反映澄清后的需求
- 是否已记录重要的澄清记录或决策
```

### **规范2: 阶段任务生成后的强制检查**
每当OrchestratorAgent为某阶段生成任务列表后，必须执行以下检查：
```markdown
【阶段任务验证检查清单】

✅ **文档完整性检查**:
- 确认当前阶段的所有必需参考文档（包括 `PROJECT_NAMING_CONVENTIONS.md`, `MODULE_INTERFACES.md` 等如适用）已存在且版本正确
- 列出任务执行需要参考的具体文档路径
- 标识缺失文档并要求补充

✅ **任务覆盖度检查**:
- 对照 `ACTION_PLAN_MASTER.md` 或分解后的 `PROJECT_OVERVIEW_DECOMPOSITION.md` 验证任务完整性
- 确保任务列表涵盖该阶段的所有关键交付物
- 检查任务间的依赖关系是否合理

✅ **质量门控设置**:
- 为每个任务明确指定必须参考的文档和验收标准
- 定义阶段完成的质量检查点，包含对重要特性的严格评估要求
```

### **规范3: 专业AI角色任务分配时的强制要求**
逻辑保持不变（参考原始文档的【AI角色任务指令模板】）。

### **规范4: UI逻辑与交互草案评审 (如适用)**
当UI设计角色提交"UI逻辑与交互草案"后，OrchestratorAgent必须：
```markdown
【UI逻辑草案评审检查清单】

✅ **需求符合性检查**:
- 草案是否准确反映了相关用户故事和功能需求
- 用户交互流程是否清晰、合理

✅ **完整性检查**:
- 是否覆盖了主要的用户场景和边缘情况
- 数据流转和界面元素定义是否初步完整

✅ **待用户/外部AI确认**:
- 明确指出此草案需要用户或外部AI进一步评审和确认
```

### **规范5: 阶段完成后的强制验证**
每个开发阶段（或重要模块交付）完成后，OrchestratorAgent必须执行：
```markdown
【阶段完成验证检查清单】

✅ **交付物完整性**:
- 检查阶段应交付的所有文档/代码/配置是否完成
- 验证文件路径是否符合项目结构规范，并符合 `PROJECT_NAMING_CONVENTIONS.md`
- 确认文档格式和内容质量，包括必要的API文档、数据模型图、测试报告等

✅ **文档一致性验证**:
- 新产出与需求文档、设计文档 (`MODULE_SPEC_[ModuleName].md` 等) 的一致性检查
- 设计文档与实现代码的对应关系验证
- 跨文档引用的准确性检查，特别是 `MODULE_INTERFACES.md` 的遵守情况

✅ **质量标准符合性**:
- 代码质量标准检查 (如适用)
- 文档完整性和可读性检查
- 测试覆盖率验证 (如适用)
- 确认重要特性已按严格标准通过测试

✅ **核心项目文档更新**:
- `AI_AGENT_PROGRESS.md` 记录阶段完成情况
- `PROJECT_CONTEXT.md` 更新关键文档索引、项目状态、技术栈等
- `PROJECT_SESSION_STATE.md` (如适用) 已更新以反映当前项目快照
- 标识发现的问题和解决方案，并更新相关文档
```

## **四、实施指导**

### **对于OrchestratorAgent**
1. 将本指南的所有检查清单作为内置行为规范，并结合 `.cursor/rules.md` 执行。
2. 在每个关键节点自动触发相应的检查流程，并整合 `custom_validation_rules.md`。
3. 强制要求专业AI角色遵循文档参考规范，并进行自我检查，**包括提交带有理由/证据的评估结果。**
4. 持续、准确地维护核心项目文档 (`PROJECT_CONTEXT.md`, `AI_AGENT_PROGRESS.md`, `PROJECT_SESSION_STATE.md` 等)。

### **对于专业AI角色**
1. 严格遵循角色特定的检查规范以及 `.cursor/rules.md` 中的通用准则。
2. 主动参考 `PROJECT_CONTEXT.md` 获取最新信息，并对自己的产出负责。
3. **在提交产出时，必须对照本指南及 `custom_validation_rules.md` 中的相关检查清单进行自我评估，并就每一项检查点的符合情况提供明确的理由或可验证的证据。**
4. 确保产出的可追溯性和可验证性。

### **对于项目用户**
1. 项目启动后可创建/修改 `custom_validation_rules.md` 添加特定要求。
2. 定期检查核心项目文档了解项目状态。
3. 在发现质量问题时及时反馈给OrchestratorAgent。
